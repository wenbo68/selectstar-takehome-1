# 중복 제거 (de-duplication)
## 정의: 중복된 수학 문제란 무엇인가?
1. 완전히 동일한 텍스트
    - 예시
        - A: "$1+1$은 무엇인가요?"
        - B: "$1+1$은 무엇인가요?"
    - 판정: 중복
2. 같은 문제; 다른 포맷팅
    - 예시
        - A: "$1+1$은 무엇인가요?"
        - B: "$1 + 1$은 무엇인가요?"
    - 판정: 중복
3. 템플릿: 변수/숫자 대체
    - 예시
        - 같은 숫자, 다른 변수
            - A: "$2x+5=7$일 때, $x$는 무엇인가요?"
            - B: "$2y+5=7$일 때, $y$는 무엇인가요?"
        - 다른 숫자, 같거나 다른 변수
            - A: "John이 5개의 사과를 가지고 있고 2개를 먹었다면, 몇 개가 남았나요?"
            - B: "Jane이 6개의 오렌지를 가지고 있고 3개를 먹었다면, 몇 개가 남았나요?"
            - A: "$2x+5=7$일 때, $x$는 무엇인가요?"
            - B: "$4x+6=10$일 때, $x$는 무엇인가요?"
    - 판정:
        - 같은 숫자, 다른 변수: 중복
        - 다른 숫자, 같거나 다른 변수: 최대 N개까지 유지
            - 예: 같은 템플릿의 예시 2개
4. 같은 개념, 다른 서술
    - 예시
        - A: "둘레가 20인 직사각형의 최대 넓이는 얼마인가요?"
        - B: "$x+y=10$이 주어졌을 때, $xy$를 최대화하세요."
    - 판정: 중복 아님
        - 같은 개념에 대한 다른 서술은 모델의 직관을 훈련하는 데 도움이 됨

## 어떻게 중복을 제거할 것인가?
1. 해싱 사용 -> 문제를 해시 값으로 변환
    1. 정확한 해싱: 같은 해시 = 중복
        - 도구
            - md5 (128bit): 더 빠름, 충돌 가능성 더 높음
            - sha-256 (256bit): 더 느림, 충돌 가능성 더 낮음
        - 처리 범위...
            - 1단계 중복: 완전히 동일한 텍스트
    2. 지역성 보존 해싱 (lsh): 유사-중복 탐지
        - 도구
            - minhash: 자카드(jaccard) 유사도 보존
        - 처리 범위...
            - 2단계 중복: 포맷팅 차이
2. 임베딩 모델 사용 -> 문제를 임베딩으로 변환
    - 모델 옵션:
        - Voyage 4
            - nDCG@10: 0.859
            - 비용: 1M 토큰당 $0.06
            - 지연 시간: 17ms
            - 차원: 1024
            - 코멘트: 가장 똑똑함; 독점적(proprietary); 권장함
        - Qwen3 Embedding 8B
            - nDCG@10: 0.818
            - 비용: 1M 토큰당 $0.05
            - 지연 시간: 56ms
            - 차원: 4096
            - 코멘트: 자체 호스팅 가능; 더 느림; 큰 임베딩으로 인해 더 많은 저장 공간 필요; 자체 호스팅용으로 권장
        - Voyage 3.5 Lite
            - nDCG@10: 0.803
            - 비용: 1M 토큰당 $0.02
            - 지연 시간: 11ms
            - 차원: 512
            - 코멘트: 매우 저렴하고 빠름; 독점적(proprietary)
    - 코사인 유사도
        - 0.99~1.0
            - 처리 범위...
                - 1단계 중복: 완전히 동일한 텍스트
                - 2단계 중복: 포맷팅 차이
            - 중복 제거
        - 0.95~0.99
            - 처리 범위...
                - 3.1단계 중복: 같은 숫자, 다른 변수
            - 중복 제거
        - 0.85~0.95
            - 처리 범위...
                - 3.2단계 중복: 다른 숫자, 같거나 다른 변수
            - 2~3개의 중복까지 유지
        - 0~0.85
            - 처리 범위...
                - 4단계 중복: 같은 개념, 다른 서술
                - 중복 아님
            - 모두 유지

## 중복 제거 향후 고려 사항
- 왜 더 복잡한 중복 제거가 필요한가?
    - 추론을 생성할 데이터가 적어짐: 더 저렴/빠름
    - 더 나은 훈련 데이터: 더 다양함
- 어떻게?
    - 수동: human in the loop (사람 개입)
        - 유사도가 0.85~0.95인 경우, 모든 유사한 문제를 가져와서 사람에게 유지할 것을 선택하도록 요청
    - 자동화: 정규화된 문제 생성
        - 유사도가 0.85~0.95인 문제의 경우, 저렴/빠른 llm을 사용하여 이름과 숫자를 변수로 하는 버전을 생성
        - 그런 다음 정규화된 문제를 임베딩하고 중복 제거

# Q/A 쌍에 추론(reasoning) 풍부하게 더하기 (enriching)
## 추론을 어떻게 생성할 것인가?
1. 추론 + 정답 생성
    - 모델 옵션: 추론에 최적화된 모델 사용
        - Gemini 3 Pro
            - artificial analysis 지능 지수: 57
            - 비용: 1M 토큰당 $4.5
            - 지연 시간: 34.07
            - 코멘트: 가장 똑똑함; 느림; 독점적(proprietary)
        - Claude Sonnet 4.6
            - artificial analysis 지능 지수: 52
            - 비용: 1M 토큰당 $6
            - 지연 시간: 0.85
            - 코멘트: 매우 빠름; 더 비쌈; 독점적(proprietary)
        - GLM-5
            - artificial analysis 지능 지수: 50
            - 비용: 1M 토큰당 $1.55
            - 지연 시간: 1.29
            - 코멘트: 저렴/빠름; 자체 호스팅 가능
    - 입력: 문제
    - 프롬프트
    - 출력: 추론 + 정답
2. 생성된 정답과 실제 정답 비교
    - 모델 옵션: 빠르고/저렴한 모델 사용
        - Gemini 2.5 Flash-Lite
            - artificial analysis 지능 지수: 19
            - 비용: 1M 토큰당 $0.17
            - 지연 시간: 0.43
            - 코멘트: 저렴/빠름; 독점적(proprietary)
        - Nova Micro
            - artificial analysis 지능 지수: 12
            - 비용: 1M 토큰당 $0.06
            - 지연 시간: 0.37
            - 코멘트: 매우 저렴/빠름; 지능 낮음; 독점적(proprietary)
        - Claude 4.5 Haiku (추론)
            - artificial analysis 지능 지수: 37
            - 비용: 1M 토큰당 $2
            - 지연 시간: 0.46
            - 코멘트: 매우 비쌈; 훨씬 높은 지능; 독점적(proprietary)
    - 입력: 생성된 정답, 실제 정답
    - 프롬프트
    - 출력: 같음/다름
        - 같음: 4단계로 이동
3. 다를 경우, 수정된 추론 생성
    - 모델 옵션: 1단계와 동일
    - 입력: 문제, 이전 추론, 이전 정답, 실제 정답
    - 프롬프트
    - 출력: 수정된 추론
4. 추론 평가 (환각, 누락된 단계, 논리적 단계)
    - 모델: 1단계와 동일
        - 교차 모델 평가: 평가를 위해 다른 모델 사용
            - 예: 1단계에서 Gemini 3 Pro를 사용한 경우, 평가를 위해 Claude Sonnet 4.6을 사용
    - 입력: 문제, 추론, 정답, 재시도 횟수(#retry)
        - 재시도 횟수(#retry) > 3, 해당 Q/A 포기 (사람의 검토를 위해 저장)
    - 프롬프트
    - 출력: 좋음/나쁨 + 나쁜 경우, 그 이유?
        - 좋음: 6단계로 이동
5. 나쁠 경우, 추론 수정
    - 모델: 1단계와 동일
    - 입력: 문제, 이전 추론, 정답, 추론이 나쁜 이유
    - 프롬프트
    - 출력: 수정된 추론
        - 4단계로 돌아감
6. 완료: 추론이 추가된 풍부해진 Q/A

# 엔드투엔드 파이프라인: 엑셀의 Q/A 쌍 -> 객체 스토리지(object storage)의 Q/A/추론
- 모든 도구 비교 (시간/비용/품질)
1. 데이터 오케스트레이션: 전체 ETL 파이프라인에 대한 스케줄/관리/모니터링
    - 도구
        - 데이터 오케스트레이션 프레임워크
            - dagster
                - 비용: 오픈소스(무료) 또는 클라우드(유료)
                - 시간: 개발 속도가 빠르지만 학습 곡선이 가파름
                - 품질: 동급 최고의 데이터 자산 추적(Software-Defined Assets)
                - 코멘트: 2026년 데이터 엔지니어링 파이프라인에서 매우 선호됨
            - prefect
                - 비용: 오픈소스(무료) 또는 클라우드(유료)
                - 시간: 기존 파이썬 함수를 작업으로 매우 빠르게 변환 (데코레이터 기반)
                - 품질: 훌륭한 작업 기반 오케스트레이션, dagster보다는 데이터 인지 능력이 약간 떨어짐
    - 왜?
        - 상태/체크포인트(state/checkpoint): 문제가 발생하면 프레임워크가 재개할 위치를 앎
        - 재시도(retry): 쉬운 재시도 데코레이터
        - 더 쉬운 배치 처리(batch processing)
        - 모니터링: ETL 관측성(observability)을 위한 내장 대시보드
    - 단계
        1. 추출(extract): 데이터 수집(ingestion)
            - 입력: 엑셀
            - 출력: DB의 Q/A 쌍
            - 도구
                - pandas/polars
                    - pandas: 중간 크기의 데이터에 충분함
                    - polars: 수백만 행에 사용; 더 나은 메모리/스레드 관리
                - 일반 DB
                    - 서버리스(serverless)
                        - 비용: 저렴; 0으로 스케일링
                        - 예: neon postgres, aws aurora serverless v2
                        - 코멘트: 이 사용 사례에 가장 적합
                    - paas
                        - 비용: 비쌈; 항상 켜져 있음
                        - 예: aws rds, google cloud sql
                        - 코멘트: 야간 배치 작업만 실행하는 경우 권장하지 않음
        2. 변환(transform): 중복 제거 + 풍부화(enrich) (추론 추가)
            - 입력: DB의 Q/A 쌍
            - 출력: DB의 Q/A + 추론
            - 도구
                - 에이전트 워크플로우 오케스트레이션
                    - langgraph
                        - 비용: 무료 (오픈 소스)
                        - 품질: 업계 표준; 복잡한 DAG에 더 좋음
                    - pydanticAI
                        - 비용: 무료 (오픈 소스)
                        - 품질: 간단함; 복잡한 DAG에는 덜 성숙함
                - 임베딩/추론 모델: 이미 비교됨
                - llm 게이트웨이
                    - litellm
                        - 비용: 무료 (자체 호스팅) 또는 저렴 (매니지드 클라우드)
                        - 시간: 모든 LLM API를 즉시 OpenAI 형식으로 표준화
                        - 품질: 신뢰할 수 있으며, 대체 라우팅 및 기본 비용 추적에 아주 좋음
                    - portkey
                        - 비용: 비쌈 (엔터프라이즈 SaaS)
                        - 품질: 프리미엄 UI, 뛰어난 견고성, 통합된 프롬프트 관리 및 관측성
                - 프롬프트 CMS
                    - llm 게이트웨이 내장
                        - litellm: 기본 프롬프트 CMS
                        - portkey: 더 고급 (A/B 테스트 등)
                    - 독립형 프롬프트 CMS
                        - promptlayer: 좋은 UI; 비기술자 사용자에게 좋음
                        - pezzo: 오픈소스
                - 벡터 DB
                    - 서버리스
                        - 비용: 저렴; 0으로 스케일링
                        - 예: pinecone serverless, neon pgvector
                        - 코멘트: 이 사용 사례에 적극 권장됨
                    - paas
                        - 비용: 비쌈; 항상 켜져 있음
                        - 예: aws rds pgvector
                - 일반 DB: 이미 비교됨
        3. 로드(load): 변환된 데이터 저장
            - 입력: DB의 Q/A/R
            - 출력: s3의 jsonl
            - 도구
                - 객체 스토리지
                    - aws s3
                        - 비용: 높은 송신(egress) 수수료
                        - 품질: 업계 표준, 모든 곳과 통합됨
                    - cloudflare r2
                        - 비용: 송신 수수료 제로
                        - 품질: S3와 호환되며 2026년에 매우 인기 있음
2. 관측성(observability)
    - ETL: dagster/prefect가 제공하는 대시보드
    - LLM (추적(Traces), 비용 추적, 평가(Eval) 지표)
        - langsmith
            - 비용: 비쌈 (엔터프라이즈)
            - 품질: LangGraph와의 기본 통합
        - langfuse
            - 비용: 무료 (자체 호스팅) 또는 저렴 (매니지드 클라우드)
            - 품질: langsmith의 인기 있는 오픈소스 대안
